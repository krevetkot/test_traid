# dm-race-detector

Модуль ядра Linux на основе фреймворка device-mapper. Создаёт виртуальные блочные устройства, которые проксируют IO-запросы на нижележащее устройство и обнаруживают гонки данных.

## Что такое гонка данных на блочном устройстве

Гонка возникает когда два IO-запроса обращаются к одним и тем же секторам одновременно, не дожидаясь завершения друг друга. Модуль проверяет два правила:

- пока не получено подтверждение записи блока X, никакая другая операция для X не может быть отправлена на устройство
- пока не получено подтверждение чтения блока X, никакая операция записи X не может быть отправлена на устройство

При обнаружении нарушения модуль выводит предупреждение в лог ядра (`dmesg`).

## Версия ядра: **Linux 6.17.0-14-generic**

## Сборка

### Требования

```bash
# Ubuntu / Debian
sudo apt install linux-headers-$(uname -r) build-essential
```

### Компиляция

```bash
make
```

После сборки в корне репозитория появится файл `dm-race-detector.ko`.

## Загрузка модуля

```bash
make load

# проверяем что модуль загрузился
dmesg | tail -3
# dm-race-detector: loaded (target name: mytarget)

# проверяем что target зарегистрирован
sudo dmsetup targets | grep mytarget
# mytarget    v0.1.0
```

## Использование

### Создание виртуального устройства

```bash
# размер в 512-байтных секторах
SIZE=2097152   # 1 GiB

# нижележащее блочное устройство
DEV=/dev/sdb

sudo dmsetup create my0 --table "0 $SIZE mytarget $DEV"
```

`/dev/mapper/my0` работает как обычное блочное устройство — все операции проксируются на `$DEV` с проверкой гонок.

### Пример с loopback-устройством

```bash
# создаём файл как backing store
dd if=/dev/zero of=/tmp/backend.img bs=1M count=1024

# создаём loop-устройство
LOOP=$(sudo losetup --find --show /tmp/backend.img)

# создаём виртуальное устройство
sudo dmsetup create my0 --table "0 2097152 mytarget $LOOP"
```

### Просмотр обнаруженных гонок

```bash
sudo dmesg | grep RACE
```

Пример вывода:

```
[23463.056766] dm_race_detector: RACE detected: WRITE [0-7] overlaps in-flight WRITE [0-7] (bio 000000005bbf7eb4)
[23463.056767] dm_race_detector: RACE detected: WRITE [0-7] overlaps in-flight WRITE [0-7] (bio 00000000aac4eacf)
```

Каждое сообщение содержит:
- тип новой операции (`READ` / `WRITE`)
- диапазон секторов новой операции
- тип и диапазон in-flight операции с которой произошла гонка
- адрес bio в памяти ядра

### Удаление устройства

```bash
sudo dmsetup remove my0
make unload
```

## Тестирование

Скрипт `test.sh` автоматически создаёт устройства, запускает тесты и выводит результат.

Для гарантированного обнаружения гонок скрипт использует `dm-delay` — стандартный модуль ядра, который добавляет задержку 300мс на каждую операцию. Это позволяет первому bio гарантированно оставаться in-flight в момент прихода второго.

```
/dev/mapper/my0     - детектор
        
/dev/mapper/delayed - dm-delay (задержка 300мс)
        
/dev/loop           - файл на диске
```

### Запуск

```bash
sudo bash ./test.sh
```

или 

```bash
make test
```

### Тесты

| # | Название | Ожидание |
|---|----------|----------|
| 1 | Write-Write overlap | RACE | 
| 2 | Read-Write overlap | RACE |
| 3 | Read-Read overlap | no RACE | 
| 4 | Non-overlapping writes | no RACE |
| 5 | Stress: 10 параллельных writes | много RACE |

## Как работает модуль

Каждый поступающий bio регистрируется в **интервальном красно-черном дереве** — структуре данных, которая позволяет за O(log n) найти все пересекающиеся диапазоны секторов.

bio поступает:
1. mytarget_map()
2. ремаппинг на нижележащее устройство
3. поиск пересечений в дереве in-flight биов
    - in-flight WRITE + любой новый  → предупреждение (Rule 1)
    - in-flight READ  + новый WRITE  → предупреждение (Rule 2)
4. вставка bio в дерево
5. передача bio дальше

bio завершается:
1. mytarget_end_io()
2. удаление bio из дерева

Дерево защищено spinlock.
